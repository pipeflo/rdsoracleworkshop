Point in time Restore

Amazon RDS for Oracle allows restore to any point in time during your backup retention period. When you restore an Oracle DB instance to a point in time, you can specify a different Oracle DB engine, license model, and DBName (SID) to be used by the new DB instance.

When performing point in time restore, there are 2 options, using the Latest Restorable Time and using Custom Time to specify the point in time restore.

Note: You can see the “Latest Restorable Time” running the following AWS CLI command:

aws rds describe-db-instances --db-instance-identifier <instance name>

Create a “Point in time Restore”

  1. On the RDS Console Database list click on your Oracle Instance to go to the Details page, then click on the Actions button and then click the Restore to point in time option.
  2. On the “Launch DB Instance” page you can configure all the option for your future restored instance.
    • On the Restore time section you can choose to restore from the Latest Restorable Time or to use a Custom Time, in this case leave Latest restorable time selected.
    • On the “Instance specifications” section, change the DB Instance Class to db.t3.small.
    • On the Settings section, enter the DB Instance Identifier name, for example pointInTimeRestore
    • On the Network & Security section, leave the selecte security group, in my case oracledb.
    • On the Database options enter PITRDB as the Database Name.
    • Leave the other field as they are and click Launch DB Instance at the bottom of the page.
  3. Finally you can see the new RDS Instance in the Databases List.

Summary

Congratulations!! You have finished the Backup and Restore Lab, where you learned how to create automated an manual backups and how to restore from them.

You can also create a restored instance from tha manual backups by clicking the Automated backups option from the RDS Console menu and then following the same steps you did on this chapter.

Other RDS Snapshots features

  • With Amazon RDS, you can copy automated or manual DB snapshots. After you copy a snapshot, the copy is a manual snapshot.
  • You can copy a snapshot within the same AWS Region, you can copy a snapshot across AWS Regions, and you can copy a snapshot across AWS accounts.
  • Copying an automated snapshot to another AWS account is a two-step process: You first create a manual snapshot from the automated snapshot, and then you copy the manual snapshot to the other account.
  • Copying Shared Snapshots
    • You can copy snapshots shared to you by other AWS accounts. If you are copying an encrypted snapshot that has been shared from another AWS account, you must have access to the KMS encryption key that was used to encrypt the snapshot.
